Materials

Quality of Service (QoS) refers to the ability of a network to provide differentiated handling and performance to various types of traffic. In modern communication systems—especially where real-time applications like voice, video, and data coexist—QoS ensures that critical services receive the bandwidth, delay, and reliability they require.

In this simulation-based project, we model a network output link shared by multiple traffic flows (VoIP, Video, and Data). Each flow generates packets at specific rates (determined by packet size and inter-arrival time) and is subjected to the selected scheduling discipline. The simulator manages packet arrivals, queueing, and transmission while computing essential QoS metrics such as throughput, delay, jitter, and packet loss.

The theoretical foundation of this system lies in queueing theory and packet scheduling algorithms. A queue represents the waiting line of packets contending for transmission on a shared link. Depending on the scheduling strategy, packets are either served in arrival order, by strict priority, or according to weighted fairness. The simulator demonstrates how these policies impact the end-to-end service quality of each traffic type.

Scheduling Algorithms Explained
  • FIFO (First-In First-Out): The simplest discipline, where packets are transmitted in the order they arrive. Although easy to implement, FIFO does not differentiate between traffic classes, meaning high-priority traffic (e.g., VoIP) may experience unacceptable delays if low-priority flows dominate the link.
  • PQ (Priority Queuing): Each traffic class is assigned a priority level. The scheduler always serves the highest-priority queue first. While this guarantees low delay for critical flows, it can cause starvation of lower-priority traffic under heavy load.
  • WFQ (Weighted Fair Queuing): A fairness-oriented scheduler that allocates bandwidth proportionally to each flow’s weight. Implemented here via Deficit Round Robin (DRR), WFQ approximates fair bandwidth distribution while accounting for variable packet sizes. This ensures balanced performance across flows with minimal starvation.

Each scheduler impacts performance differently. For instance, VoIP requires low delay (<150 ms) and low jitter (<30 ms), Video demands higher throughput but moderate delay tolerance, and Data transfers (like FTP or downloads) can tolerate higher delay but require reliability. The simulator evaluates whether each flow meets its QoS thresholds based on measured metrics.

How We Built This

The QoS Parameter Analyzer was developed as an educational tool to visualize and analyze the impact of scheduling on network performance. The interface follows a modern, Fluent-inspired design with accessibility and responsiveness as key principles.

  1. Designed a clean and responsive user interface using HTML5, CSS3 (Flex & Grid), and accessible semantics for screen readers.
  2. Implemented a discrete-event simulation engine to handle packet generation, queueing, and service under FIFO, PQ, and DRR-based WFQ scheduling policies.
  3. Each packet event tracks timestamps for arrival, queueing delay, and departure to compute precise QoS metrics.
  4. Visualized performance results (throughput, delay, jitter, loss) using the Canvas 2D API with threshold-based coloring to show compliance.
  5. Enabled users to download detailed reports (in pdf) summarizing simulation parameters and computed metrics.
  6. Ensured cross-device compatibility and keyboard navigation to support all learners effectively.

Help

How to use

  1. Set link capacity, buffer, duration at the top.
  2. Add flows (VoIP/Video/Data…) and set packet size (bytes) and inter-arrival (ms). For PQ, set priority (1=High). For WFQ, set weights.
  3. Define QoS thresholds per flow (delay, jitter, loss).
  4. Select Scheduling Policy and click Simulate.
  5. Review the Results table and charts to see compliance.
  6. Click Download to export inputs, outputs, and the step-by-step procedure.

Example inputs

  • VoIP: 160 B, inter-arrival 20 ms, priority 1, weight 5; thresholds: delay ≤ 150 ms, jitter ≤ 30 ms, loss ≤ 1%
  • Video: 1200 B, inter-arrival 10 ms, priority 2, weight 3; thresholds: delay ≤ 200 ms, jitter ≤ 50 ms, loss ≤ 1%
  • Data: 1500 B, inter-arrival 40 ms, priority 3, weight 1; thresholds: delay ≤ 1000 ms, jitter ≤ 100 ms, loss ≤ 5%

Execution flow

The simulator generates packet arrivals for each flow until the duration elapses, enqueues them subject to a shared buffer, schedules them based on the selected policy, and records completion times.

What is processed

  • Service time per packet = size (bits) / link rate (bps)
  • Queueing delay accumulates when the link is busy
  • Drops occur when total enqueued packets exceed buffer size

Interpret results

  • Throughput (Mbps): delivered bits / total simulation time
  • Delay (ms): mean end-to-end per flow
  • Jitter (ms): mean absolute difference between consecutive packet delays
  • Loss (%): (dropped / total) × 100
  • QoS pill: Green = meets all thresholds; Red = one or more violations

What the charts depict

  • Throughput bars per flow (higher is better)
  • Delay, Jitter, Loss bars with threshold lines (green if under threshold)
